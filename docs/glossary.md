# Glossary

Terms used throughout this guide, explained in plain language. If you arrive on a page mid-guide and encounter an unfamiliar term, this is your reference.

---

ADC (Application Default Credentials)
:   Google's mechanism for finding credentials automatically based on the environment your code is running in. GCP client libraries (`bigrquery`, `googleCloudStorageR`, `google-cloud-bigquery`) use ADC to authenticate without requiring credentials to be written into code.

Artifact Registry
:   A GCP service that stores Docker images. The base images (`gcp-etl`, `gcp-app`) are pushed here by GitHub Actions after being built, and pulled from here when Cloud Run Jobs start.

BigQuery
:   Google's cloud data warehouse. Pipelines read source data from and write results to BigQuery tables using SQL. BigQuery scales to handle billions of rows without requiring you to manage any infrastructure.

branch
:   A parallel line of development in Git. You create a branch to make changes in isolation from `main`, then merge it back via a pull request once the work is reviewed and tests pass.

branch protection
:   A GitHub setting that prevents anyone from pushing directly to `main`. All changes must arrive through a pull request that has passed automated tests and been approved by at least one reviewer.

CI/CD (Continuous Integration / Continuous Deployment)
:   The practice of running tests automatically on every proposed code change (CI) and deploying automatically when changes are merged (CD). GitHub Actions implements both in this workflow.

Cloud Run Job
:   A GCP service that starts a container, runs it to completion, and stops it. Used for ETL pipelines and scheduled scripts. You are billed only for the time the container is actually running — there is no idle cost.

Cloud Scheduler
:   A GCP managed cron service. Triggers Cloud Run Jobs on a time-based schedule (for example, "run at 07:00 every Monday"). Configured by the platform team from your `cloud-run-job.yml` file.

container
:   A running instance of a Docker image. Containers are isolated, consistent, and disposable — starting a container always gives you the same environment, regardless of what machine it is running on.

ETL (Extract, Transform, Load)
:   The standard pattern for data pipelines: extract data from a source, apply transformations, then load the results to a destination. In this guide, `src/extract.R`, `src/transform.py`, and `src/load.R` implement each phase.

gcsfuse
:   A tool that makes a Google Cloud Storage bucket accessible as a regular filesystem directory. Used by Cloud Run Jobs to mount pipeline code from a GCS bucket at `/workspace` inside the container.

GCS (Google Cloud Storage)
:   Google's object storage service — conceptually similar to a shared network drive, but accessible from anywhere with the right IAM permissions. Used to store pipeline code and pipeline output files.

IAM (Identity and Access Management)
:   GCP's permission system. Controls which service accounts and users can access which resources. Follows the principle of least privilege — each pipeline service account has only the permissions it needs and no more.

image
:   A Docker image is a complete, layered snapshot of an environment: operating system, language runtime, system libraries, and packages. A container is a running instance of an image.

main branch
:   The definitive, production-ready version of the code. Branch protection ensures nothing reaches `main` without review and passing tests. When a pipeline syncs to GCS on merge to `main`, the next Cloud Run execution uses the new code.

pull request (PR)
:   A GitHub mechanism for proposing changes. A pull request shows the exact diff, triggers automated tests, and provides a structured, permanent record of code review. Merging a pull request is the only way changes reach `main`.

RAP (Reproducible Analytical Pipeline)
:   A UK Government Analysis Function framework for modernising analytical workflows. Pipelines following RAP principles are version-controlled, containerised, automatically tested, and auditable. This guide implements RAP using Git, Docker, and GCP.

renv
:   An R package for reproducible environments. Records the exact version, source, and hash of every installed R package in `renv.lock`. Used during Docker builds to ensure every machine and every Cloud Run Job installs exactly the same packages.

renv.lock
:   A JSON file generated by `renv::snapshot()` that records the exact version, source, and integrity hash of every R package in the project. Committed to Git so that `renv::restore()` during Docker builds produces a byte-for-byte identical package set.

Secret Manager
:   A GCP service that stores sensitive configuration values (project IDs, bucket names, API keys) in encrypted storage and provides permissioned, audited access to them. Cloud Run Jobs retrieve secrets from Secret Manager and inject them as environment variables at runtime.

Service Account
:   A GCP identity for applications rather than people. The Cloud Run Job runs as a service account with specific IAM roles that determine what GCP resources it can read or write. Configured by the platform team.

special category data
:   The UK GDPR term (Article 9) for particularly sensitive personal data: health records, ethnicity, biometrics, religious beliefs, and similar. Requires stronger justification for processing than ordinary personal data.

WIF (Workload Identity Federation)
:   A mechanism that allows GitHub Actions to authenticate to GCP without storing a long-lived service account key as a GitHub Secret. More secure than key-based authentication because there is no persistent credential to leak.

WSL2 (Windows Subsystem for Linux 2)
:   A Microsoft-provided feature of Windows 10/11 that runs a genuine Linux kernel inside a lightweight virtual machine on your Windows laptop. Provides a full Linux development environment — including Docker, git, and gcloud — without replacing Windows.

`/workspace`
:   The standard directory inside Docker containers where pipeline code runs. Locally, your project folder is mounted here via `docker-compose.yml`. In Cloud Run, a GCS bucket subfolder is mounted here via `cloud-run-job.yml`. Scripts always reference `/workspace/src/...` without needing to know which mount mechanism is in use.
